Review

Topics Covered: Rails relations; associating two models;

Exercises

E1: Relation Review

  1. Why do we use relations? What are they good for?
  2. Will you ever write a relation on just one class, or will a relation always join two classes together?
  3. What types of relations did we learn about? What makes them unique? Can you give examples of each? Hint: We learned about 4 different kinds and an extension.
  4. What changes do you have to make to migrations for each relation? Does every single one require making changes to the database?

E2: We Belong Together

What kind of relations would you use for each of the following models?

  1. Tweets and users
  2. Users and accounts
  3. Tweets and hash tags
  4. Stores and customers
  5. Shopping carts and items
  6. Shoppers and Carts
  7. Shopper and items in the cart that the shopper has

E3: Omg Really Twitter Again?!

Rebuild Twitter. Create Tweet and User from scratch, without using scaffolding. Make Tweet and User related in a way that makes sense.

E4: #relationships

In the app you built above, add a HashTag resource, complete with a migration and a model (no need for routes, controllers, or views quite yet). Make it associated with Tweet in a has_and_belongs_to_many relationship. Don't forget to give it a join table!

Adding content via Terminal

Next, modify User to have many hash tags through tweets. Since we don't know how to add hash tags to a tweet via our website yet, create some users and give them tweets in the rails console. Then make some hash tags, and add them to those tweets. Make sure user.hash_tags returns an array of HashTags.

Show Me the Tags

Write a route, controller action and view to show all the hash tags a user has ever used in all of their tweets. It should really not be too much code—most of the heavy lifting is done by the Rails relations!
